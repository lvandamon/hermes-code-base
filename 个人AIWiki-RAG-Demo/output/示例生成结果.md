# 文章标题：RAG 系统不只是向量检索

## 1. 一页中文摘要

这篇文章指出，很多初学者会把 RAG（Retrieval-Augmented Generation，检索增强生成）简单理解为“文档切块、Embedding（嵌入）、向量相似度搜索”，但这只是 RAG 系统的最基础形态。真实生产环境中的 RAG 更像一条围绕数据处理、检索、排序、上下文组织、生成和评估构成的工程闭环。

文章强调，Chunking（切分）会影响信息完整性，Embedding（嵌入）会影响语义召回，Retriever（检索器）决定候选内容范围，Rerank（重排序）决定哪些内容最终进入 Prompt（提示词）上下文。如果只依赖向量相似度，系统可能召回“语义相近但不能回答问题”的片段，也可能漏掉关键词高度相关的事实。因此，Hybrid Search（混合检索）、Query Rewrite（查询改写）和 Multi-hop Retrieval（多跳检索）经常用于提升检索质量。

文章还提醒，RAG 优化不能只凭感觉，需要 Evaluation（评估）。检索阶段可以关注 Recall（召回率）、Precision（准确率）、MRR（平均倒数排名）、NDCG（归一化折损累计增益）；生成阶段可以关注 Faithfulness（忠实性）、Answer Relevance（答案相关性）和 Citation Accuracy（引用准确性）。对我学习 RAG / AI 工程的启发是：不要把 RAG 项目包装成“接入向量数据库”，而要能讲清楚从数据、检索、排序到生成评估的完整闭环。

## 2. 术语中英文对照

| 英文术语 | 中文翻译 | 在本文中的含义 | 我需要记住的重点 |
|---|---|---|---|
| RAG | 检索增强生成 | 通过检索外部知识增强大模型回答 | RAG 是系统工程，不只是向量搜索 |
| Embedding | 嵌入 | 把文本映射成向量表示 | 影响语义召回质量 |
| Chunking | 切分 | 把长文档切成适合检索的小片段 | 切分粒度会影响上下文完整性 |
| Retriever | 检索器 | 从知识库召回候选片段的组件 | 决定候选文档集合 |
| Rerank | 重排序 | 对候选片段重新排序 | 决定最终进入上下文的内容 |
| Hybrid Search | 混合检索 | 结合向量检索和关键词检索 | 可弥补单一检索方式缺陷 |
| Query Rewrite | 查询改写 | 把用户问题改写为更适合检索的查询 | 适合复杂或模糊问题 |
| Evaluation | 评估 | 衡量检索和生成质量 | 没有评估就无法稳定优化 |

## 3. 关键概念表

| 概念 | 解释 | 与 RAG 的关系 | 典型应用/例子 | 易混点 |
|---|---|---|---|---|
| 向量检索 | 根据向量相似度召回文本片段 | RAG 的基础检索方式之一 | 语义搜索、FAQ 检索 | 不等于完整 RAG 系统 |
| 混合检索 | 同时使用语义与关键词信号 | 提高召回覆盖率 | BM25 + Vector Search | 不是简单把两个结果拼接 |
| 重排序 | 对候选结果二次排序 | 提升进入上下文内容的质量 | Cross-Encoder Reranker | 和初次召回不是同一阶段 |
| 检索评估 | 衡量召回结果是否命中答案 | 帮助定位检索问题 | Recall、MRR、NDCG | 不应只看最终回答好坏 |
| 生成评估 | 衡量回答是否忠实、相关、可引用 | 帮助判断答案质量 | Faithfulness、Citation Accuracy | 生成好不代表检索好 |
